# 用户逻辑：Racingline 策略研究工作台

状态：当前用户画像草案（2026-06-20）

路径说明：本文写于 Plan 0053 迁移前；文中的 `app/racingline_new/` 为历史原型路径，当前 Racingline 前端代码根为 `app/racingline/`。

## 用户画像

我是一个使用数据辅助决策的 A 股策略研究者。我的目标不是在系统里直接下单，而是把一个选股想法变成可校验、可追溯、可复盘的研究型策略。

我需要的不是“行情软件”或“券商账户”，而是一个内部研究工作台：

- 用结构化规则表达选股逻辑。
- 用 Rearview 在历史交易日上生成股票池和买入信号。
- 用研究型虚拟账户把信号转成组合运行结果。
- 用净值、持仓、成交、费用和风险指标判断策略是否值得继续观察。
- 每天盘后回到看板，快速知道哪些策略、信号或异常需要处理。

## 当前产品定位

Racingline 是 Rearview 的前端研究工作台。它负责把策略研究流程组织成可操作 UI；权威规则校验、选股执行、组合运行、成交假设、持仓账本和净值计算由 Rearview、worker 和数据平台承担。

当前边界：

- Racingline 不直接访问 ClickHouse 或 PostgreSQL。
- Racingline 不在浏览器内重算技术指标、选股结果、成交、费用、持仓或净值。
- 系统不接入真实券商账户，不做实盘下单、撤单或清算。
- 虚拟账户是研究参数模板，不是真实资金账户。
- 组合净值是研究型结果，用于比较策略，不代表真实券商收益。

## 使用时序

系统的主要使用场景是盘后研究：

1. 数据平台完成行情和指标更新。
2. Rearview 根据规则版本生成选股 run、股票池和买入信号。
3. 组合 worker 根据信号、账户快照和执行规则生成组合净值与明细账本。
4. 用户打开 Racingline，看今天哪些策略有信号、哪些组合表现异常、哪些策略需要继续调整。

> 核心节奏是盘后复盘和策略研究，不是盘中盯盘，也不是自动交易。

## 核心闭环

### 1. 看板判断今天先看什么

我打开看板，首先看到正在观察的策略组合。每个策略卡片需要告诉我：

- 策略名称和运行状态。
- 建仓日期和运行天数。
- 最新净值和近期变化。
- 收益与风险摘要。
- 今日信号股票及得分。
- 净值与基准走势。
- 是否有失败运行、缺失价格、现金不足、规则异常等 warning。

看板要回答一句话：

> 今天我应该优先关注哪个策略、哪只股票、哪个异常？

### 2. 进入策略详情追查原因

当某个策略卡片显示新的今日信号或异常表现时，我进入策略详情。详情页需要让我看到：

- 历史信号数量和某日信号股票列表。
- 当前持仓、持仓收益、持仓天数和贡献。
- 调仓记录、买入/卖出/持有原因。
- 策略绩效摘要和净值曲线。
- 从策略信号或持仓继续进入个股分析页的路径。

详情页要回答一句话：

> 这个策略今天为什么动、当前持有什么、结果是否符合预期？

### 3. 创建或修改策略规则

当我有新的策略想法，或发现旧策略需要调整时，我进入策略创建/编辑流程。我希望用表单和组件表达规则，而不是手写 SQL 或任意代码。

规则创建流程应支持：

- 选择可用指标。
- 配置过滤条件组。
- 配置评分权重。
- 通过 explain 或预览理解依赖指标、输出列和可能命中的股票。
- 发布不可变规则版本，保证历史 run 可复现。

策略创建页要回答一句话：

> 这个选股逻辑是否能被系统稳定表达和校验？

### 4. 预览股池再决定是否继续

在提交完整运行前，我需要先看当前规则大概会命中哪些股票。

股池预览应展示：

- 股票名称和证券代码。
- 排名或得分。
- 命中的关键指标或得分项。
- 当前条件过宽、过窄或数据缺失时的提示。

预览要回答一句话：

> 这组规则选出来的股票是否像我以为的那样？

### 5. 配置研究型虚拟账户

选股规则只是信号来源。我要判断策略能不能运行，还需要把信号转成可研究的组合。

虚拟账户配置应覆盖：

- 初始资金，默认 `1,000,000 CNY`。
- 佣金、印花税、过户费和滑点。
- TopN、目标权重和仓位约束。
- 止盈、止损、时间止损或清仓规则。
- 默认 T 日收盘后得到信号，T+1 交易日成交，避免未来函数。

虚拟账户配置要回答一句话：

> 在明确成交假设和费用口径下，这个策略会变成怎样的组合？

### 6. 回测或组合运行验证

当规则和账户参数确认后，我发起回测或组合运行。系统异步计算，不在浏览器请求里同步跑完整净值。

结果需要展示：

- 运行状态和错误原因。
- 净值曲线、累计收益、日收益和最大回撤。
- 现金、持仓市值、总资产和仓位比例。
- 费用与滑点成本汇总。
- 每日目标组合、虚拟订单、虚拟成交和持仓明细。
- 与规则版本、选股 run 和账户快照的可追溯链接。

组合运行页要回答一句话：

> 这个策略按当前规则和账户假设运行后，结果是否值得继续观察？

### 7. 运行策略并回到看板观察

当回测或组合运行结果可以接受时，我把策略加入持续观察。后续每天盘后，系统继续生成信号和组合结果，我回到看板查看变化。

运行策略不是实盘交易，而是进入研究观察状态：

- 后续信号继续由 Rearview 生成。
- 后续组合结果继续由虚拟账户和 worker 计算。
- 看板负责暴露今日信号、净值变化和异常。
- 详情页负责解释持仓、调仓和策略表现。

这一环节要回答一句话：

> 这个策略是否已经从一次性实验变成可持续复盘的研究对象？

## 当前原型对应关系

`app/racingline_new/` 当前原型验证的是下面的主路径：

```text
看板
  -> 策略详情
  -> 选股/策略创建
  -> 权重配置
  -> 股池预览
  -> 模拟建仓
  -> 回测
  -> 运行策略
  -> 回到看板
```

该原型使用 mock 数据验证页面职责和信息层级。正式实现时，mock 数据必须迁移到 Rearview API、组合运行 API 和后端持久化账本。

## 不在当前阶段做的事

- 实盘下单和券商账户接入。
- 登录、权限、多用户资金隔离和审计合规。
- 任意 SQL、Python、Rust 或前端公式作为用户自定义策略逻辑。
- 浏览器内权威计算技术指标、成交、持仓、费用或净值。
- 分钟级撮合、盘口排队、部分成交、融资融券、期权、期货、多币种和真实企业行动清算。

## 产品原则

1. **研究闭环优先**：页面应围绕“想法 -> 规则 -> 信号 -> 组合 -> 复盘”组织，而不是围绕功能清单组织。
2. **权威计算后移**：浏览器负责表达、展示和操作；规则、信号、成交、持仓和净值由后端和数据平台负责。
3. **历史可复现**：规则版本、账户快照、执行参数和价格口径必须可追溯，不能被后续编辑 silently 改写。
4. **信号不是交易指令**：买入信号是研究入口，不是自动下单指令。
5. **异常必须可见**：价格缺失、现金不足、跳过买入、失败运行和数据口径限制都应在 UI 中可解释。
6. **默认防未来函数**：默认使用 T 日信号、T+1 成交；任何实验性 same-day 模式都必须明确标记。
7. **从看板回到看板**：用户每天盘后从看板开始，也应在创建或调整策略后回到看板持续观察。
