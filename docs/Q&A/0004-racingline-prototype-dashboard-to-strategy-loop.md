# Q&A 0004: Racingline 原型看板到策略创建闭环用户故事

状态：Proposed

日期：2026-06-20

路径说明：本文写于 Plan 0053 迁移前；文中的 `app/racingline_new/` 为历史原型路径，当前 Racingline 前端代码根为 `app/racingline/`。

## 背景

`app/racingline_new/` 当前可视为 Racingline 的快速交互原型，用于验证一个从盘后看板出发，到选股、回测，再把策略投入运行的业务闭环。该原型不是生产实现，也不代表最终 API contract；它的价值在于把用户主路径、页面职责和信息层级先跑通。

当前原型路由：

| 路由 | 原型职责 |
|---|---|
| `/dashboard` | 策略看板，总览组合表现、运行天数、今日信号、风险收益指标和净值曲线 |
| `/dashboard/strategies/:portfolioId` | 策略详情，查看策略信号、策略业绩、当前持仓和调仓记录 |
| `/strategies` | 策略创建/编辑流程，覆盖选股、权重、股池预览、模拟建仓和回测 |

## 用户故事

作为一个盘后复盘的 A 股策略研究者，我希望先在看板上快速判断哪些策略今天值得关注，再进入选股和回测流程验证新的策略想法，最后把通过验证的策略创建为可运行对象。这样我可以把投资想法从“临时判断”推进到“可追踪、可复盘、可持续观察”的研究型策略。

## 主流程

1. 用户打开 `/dashboard`，先看到已有策略的运行概况。
2. 看板卡片展示策略名称、建仓日期、运行天数、最新净值、收益指标、风险指标、今日信号和净值曲线。
3. 用户从今日信号中识别值得研究的股票，必要时进入 `/dashboard/strategies/:portfolioId` 查看该策略的信号历史、持仓和调仓记录。
4. 用户点击看板中的 `选股` 或创建入口，进入 `/strategies`。
5. 用户在策略创建流程中配置指标条件，进入权重配置，再用股池预览检查当前规则命中了哪些股票。
6. 用户配置模拟建仓参数，包括初始资金、买入 TopN、仓位限制、交易成本和卖出/止损规则。
7. 用户执行回测，检查收益、风险、净值曲线、持仓记录和交易记录。
8. 回测结果可以接受时，用户点击 `运行策略` 返回看板，策略进入后续盘后观察和复盘入口。

## 原型已验证的页面职责

### 看板

看板不是营销首页，而是盘后工作台。它应优先回答：

> 今天哪些策略值得我先看？

当前原型支持的关键元素：

- 策略卡片按组合展示，而不是按功能目录展示。
- 卡片顶部突出策略名称、建仓日期、运行天数和最新净值。
- `收益指标` 和 `风险指标` 保留为首要绩效摘要。
- `今日信号` 在风险指标之后展示股票名称、证券代码和得分；超过 5 条滚动，不足 5 条用 `--` 占位。
- `净值与基准` 放在卡片下部，用于快速判断策略走势。
- `选股` 是从看板进入策略创建流程的主动作。

### 策略创建流程

策略创建页不是单纯表单，而是一个逐步收敛的研究流程。它应优先回答：

> 这个策略怎么写、命中了什么、历史表现是否值得运行？

当前原型支持的步骤：

- `策略选股`：配置指标条件组。
- `权重配置`：配置评分权重。
- `股池预览`：查看当前规则命中的股票。
- `模拟建仓`：配置资金、仓位、交易成本和卖出/止损参数。
- `策略回测`：查看绩效指标、净值曲线、持仓和交易记录。
- `运行策略`：把通过回测的策略带回看板观察。

## 原型边界

- 原型使用本地 mock 数据验证工作流，不代表最终数据来源。
- 原型不直接访问 ClickHouse 或 PostgreSQL。
- 原型不在浏览器内实现权威选股、成交、持仓、费用、滑点或净值计算。
- `运行策略` 当前是交互闭环占位，不代表后端已经完成策略创建、调度或组合运行状态机。
- 真实生产实现需要通过 Rearview API、portfolio data plane 和虚拟账户/回测 worker 接管权威状态。

## 后续沉淀要求

当该原型进入正式实现计划时，需要把以下事项写入 RFC 或 plan：

1. `/dashboard` 卡片数据的 Rearview API contract。
2. 今日信号、策略详情信号列表和股池预览之间的共享数据结构。
3. `运行策略` 对应的后端创建、回测结果绑定、虚拟账户启动和状态回写流程。
4. mock 数据到真实 API 的迁移策略。
5. 浏览器验收范围：桌面首屏、窄屏、信号滚动、空状态、回测完成态和返回看板路径。
