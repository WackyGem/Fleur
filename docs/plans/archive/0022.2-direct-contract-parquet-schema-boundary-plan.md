## 1. 背景

Plan 0022 已把 `pipeline/contracts/datasets/*.yml` 的 `parquet.fields` 收敛为 Dagster source assets 写 S3 Parquet 的 schema 事实源，并新增了 `fleur_contracts.adapters.parquet`。但 Plan 0022 同时引入了 scheduler checked-in generated Parquet schema module：

```text
pipeline/contracts/datasets/*.yml
  -> fleur_contracts.adapters.parquet
  -> pipeline/scheduler/scripts/generate_parquet_schemas.py
  -> pipeline/scheduler/src/scheduler/defs/generated/parquet_schemas.py
  -> scheduler runtime
```

用户指出该方案在 scheduler 内形成了与 ClickHouse raw specs 不一致的 contract 消费模式。当前 ClickHouse raw specs 已经在 scheduler definitions 层直接通过 `fleur_contracts.loader.load_registry()` 和 `fleur_contracts.adapters.clickhouse` 派生，而 Parquet schemas 则通过 checked-in generated Python 文件分发。

本计划一刀切清理该不一致：删除 scheduler Parquet schema checked-in generated 链路，改为 scheduler 内单一 contract boundary module 直接 import `fleur_contracts.adapters.parquet`，与 ClickHouse raw specs 的 direct import 模式保持一致。

## 2. 目标

完成后应满足：

1. `pipeline/contracts/datasets/*.yml` 仍是 `source -> S3 Parquet -> ClickHouse raw` 的唯一字段事实源。
2. `pipeline/contract_tools/src/fleur_contracts/adapters/parquet.py` 继续作为唯一 PyArrow type parser、Parquet schema contract 和 schema hash adapter。
3. 删除 `pipeline/scheduler/scripts/generate_parquet_schemas.py`。
4. 删除 `pipeline/scheduler/src/scheduler/defs/generated/parquet_schemas.py` 和只为该文件存在的 `pipeline/scheduler/src/scheduler/defs/generated/__init__.py`。
5. 新增 scheduler 集中 contract boundary module，例如 `pipeline/scheduler/src/scheduler/defs/contract_schemas.py`，由它直接从 `fleur_contracts.loader.load_registry()` 和 `fleur_contracts.adapters.parquet` 派生：
   - dataset -> `pa.Schema`
   - dataset -> `parquet_schema_hash`
   - dataset -> `contract_schema_hash`
   - dataset -> `source_schema_hash`
   - dataset -> source field names
   - dataset -> contract version
   - dataset -> source asset key
   - dataset -> storage mode
   - dataset -> partition key name
6. scheduler source business code、`S3IOManager`、compact helpers 和 EastMoney wrapper 不再 import `scheduler.defs.generated.parquet_schemas`，统一从 scheduler contract boundary module 读取 schema/maps。
7. scheduler 允许在明确的 boundary modules 直接 import `fleur_contracts`：
   - 已存在：`scheduler.defs.clickhouse.specs`
   - 新增：`scheduler.defs.contract_schemas`
8. source business code 仍不得自行解析 YAML 或直接 import `fleur_contracts`。
9. `S3IOManager` schema equality guard 和 contract/schema hash metadata 行为不退化。
10. 删除 Parquet generated currentness 测试，改为 direct contract boundary coverage 测试。

## 3. 非目标

本计划不做以下事情：

1. 不改变 `pipeline/contracts/datasets/*.yml` 的字段事实结构。
2. 不改变 S3 object layout、partition 目录规则或 ClickHouse raw sync staging/replace 协议。
3. 不迁移 dbt staging 字段事实到 contracts。
4. 不恢复 scheduler source modules 中的 dataset-level 手写 `pa.schema(...)`。
5. 不保留 `scheduler.defs.generated.parquet_schemas` 作为 fallback。
6. 不实现 IO manager 自动 cast；source conversion 仍必须产出正确 typed table。
7. 不保留 EastMoney OpenAPI 字段顺序生成链路；EastMoney 字段顺序来自 contract `source.fields`。
8. 不在本计划内重构 ClickHouse raw specs；只把 Parquet schema 消费模式改到与 ClickHouse 当前模式一致。

## 5. 目标架构

改造后 schema 链路为：

```text
pipeline/contracts/datasets/*.yml
  -> fleur_contracts.loader.load_registry()
  -> fleur_contracts.adapters.parquet
  -> scheduler.defs.contract_schemas
  -> source conversion functions
  -> S3IOManager schema equality check
  -> S3 Parquet
  -> ClickHouse raw sync specs
```

边界要求：

| 层 | 允许读取 contracts / import fleur_contracts | 说明 |
|----|--------------------------------------------|------|
| `contract_tools` | 是 | contract parser、adapter、hash、validate/generate |
| `scheduler.defs.contract_schemas` | 是 | Parquet schema/maps 的 scheduler contract boundary |
| `scheduler.defs.clickhouse.specs` | 是 | 已存在 ClickHouse raw sync spec boundary |
| scheduler source business code | 否 | 可 import `scheduler.defs.contract_schemas` 暴露的 schema/maps，不直接 import `fleur_contracts` |
| `S3IOManager` | 否 | 不直接 import `fleur_contracts`；只 import scheduler boundary module |
| tests | 是 | 可直接 import `fleur_contracts` 做覆盖性断言 |
